❓ Como Usar

Guia de referência rápida para os formatos suportados.

📚 Formatos Suportados

  • Valgrind Memcheck — Gera um .log com erros de memória detalhados e stack traces.
  • AddressSanitizer (ASan) — Compile com -fsanitize=address e redirecione o stderr para um arquivo.

🔧 Comando Recomendado — Valgrind

valgrind \
  --tool=memcheck \
  --leak-check=full \
  --show-leak-kinds=all \
  --track-origins=yes \
  --error-exitcode=1 \
  ./server 2> valgrind.log

🔧 Comando Recomendado — ASan

cmake .. -DCMAKE_CXX_FLAGS="-fsanitize=address -g" \
         -DCMAKE_EXE_LINKER_FLAGS="-fsanitize=address"
ASAN_OPTIONS=log_path=asan.log ./server

🎯 Classificação de Severidade

  • 🔴 CRITICAL — Invalid write / use-after-free write: corrompe heap, pode causar crash imprevisível.
  • 🟡 HIGH — Invalid read: lê memória já liberada, comportamento indefinido.
  • ⚠️ WARNING — Use of uninitialised value / cond jump: lógica incorreta.
  • 💧 LEAK (definitely lost) — Memória alocada e nunca liberada.
  • 🔵 LEAK (possibly / indirectly) — Provável não-problema em processos de longa execução.

💡 O que priorizar primeiro

  • ❌ Invalid write em bloco free'd → crash imprevisível + corrupção de dados.
  • ❌ use-after-free → crash em operações com objetos já destruídos.
  • ❌ heap-buffer-overflow → leitura/escrita fora dos limites alocados.
  • ⚠️ Definitely lost crescendo → memory leak real que precisa ser corrigido.

📖 Guia Valgrind & ASAN

Guia completo: do build à leitura do log e correção de leaks no TFS.

📌 Visão Geral

Memory leaks no TFS causam crescimento contínuo do RSS, degradando performance e podendo derrubar o servidor após horas de uptime.

🛡️

Valgrind Memcheck

Detecta leaks com backtrace completo. Sem recompilação especial. 10-50× mais lento.

⚡

AddressSanitizer (ASAN)

Instrumentação em compile-time. ~2× overhead. Detecta use-after-free, stack/heap overflow, leaks.

🔎

LeakSanitizer (LSAN)

Integrado ao ASAN. Relatório detalhado de leaks no encerramento do processo.

🎯

Estratégia Recomendada

ASAN no desenvolvimento (CI/testes), Valgrind para análise profunda de leak específico.

🔍 Tipos de Leak (Valgrind)

Valgrind classifica memória não liberada em categorias. Entender cada uma determina a urgência:

Definitely Lost CRÍTICO

Nenhum ponteiro aponta para o bloco. Memória 100% irrecuperável. Sempre corrija primeiro.

Indirectly Lost ALTO

Bloco apontado por outro que é "definitely lost". Corrija o pai e este some automaticamente.

Possibly Lost MÉDIO

Ponteiro aponta para o interior do bloco. Pode ser intencional (pointer arithmetic) ou bug.

Still Reachable BAIXO

Ponteiro válido existe, mas free() nunca foi chamado. Comum em singletons/globals. Frequentemente ignorável.

⚠️
Foco no TFS

Priorize Definitely Lost e Indirectly Lost. "Still Reachable" em singletons como g_game, g_config é normal.

🔨 Build para Debug

Para backtraces úteis, compile com símbolos de debug (-g) e sem otimizações:

# Na raiz do TFS mkdir -p build-debug && cd build-debug cmake .. -DCMAKE_BUILD_TYPE=Debug \ -DCMAKE_CXX_FLAGS="-g -O0 -fno-omit-frame-pointer" \ -DCMAKE_LINKER_FLAGS="-rdynamic" cmake --build . -- -j4
💡
Por que -fno-omit-frame-pointer?

Garante que o frame pointer seja mantido na stack, permitindo backtraces completos e precisos pelo Valgrind/ASAN.

🛡️ Valgrind Memcheck — Comandos

Comando definitivo para capturar apenas leaks reais (definite) com máximo detalhe:

🎯 Apenas leaks definitivos

valgrind \ --tool=memcheck \ --leak-check=full \ --show-leak-kinds=definite \ --track-origins=yes \ --num-callers=50 \ --leak-resolution=high \ --error-limit=no \ --expensive-definedness-checks=yes \ --partial-loads-ok=yes \ --log-file=valgrind-definitive.log \ ./tfs

📋 Todos os tipos de leak

valgrind \ --tool=memcheck \ --leak-check=full \ --show-leak-kinds=all \ --track-origins=yes \ --num-callers=50 \ --leak-resolution=high \ --error-limit=no \ --suppressions=tfs.supp \ --log-file=valgrind-full.log \ ./tfs

🔥 Detecção de use-after-free

valgrind \ --tool=memcheck \ --leak-check=full \ --show-leak-kinds=definite,indirect \ --track-origins=yes \ --num-callers=50 \ --undef-value-errors=yes \ --malloc-fill=0xAA \ --free-fill=0xDD \ --error-limit=no \ --log-file=valgrind-uaf.log \ ./tfs
🐢
Performance

Valgrind executa o TFS 10–50× mais devagar. Desative scripts pesados no config.lua. Use mapa reduzido com poucos NPCs.

🔧 Scripts para Resumo Rápido

# Resumo do leak summary grep -E "(definitely|indirectly|possibly|reachable|suppressed):" valgrind.log | tail -10 # Top backtraces de leaks definitivos grep -B1 -A 25 "definitely lost" valgrind.log | head -500 # Listar arquivos .cpp envolvidos nos leaks grep -oP "[a-zA-Z_/]+\.cpp:\d+" valgrind.log | sort | uniq -c | sort -rn | head -20

⚡ ASAN + LeakSanitizer — Build & Execução

ASAN é integrado ao GCC/Clang. Detecta use-after-free, buffer overflow e leaks. Muito mais rápido que Valgrind.

💀

Use-After-Free

Acesso a memória já liberada. Causa crashes ou UB silencioso.

📦

Heap Buffer Overflow

Escrita além do final de um buffer alocado no heap.

📚

Stack Buffer Overflow

Overflow em variáveis locais. Pode causar RCE.

🔗

Memory Leaks (LSAN)

Detectados ao encerrar o processo. Similar ao Valgrind.

🔧 Opção 1: Flags direto no cmake

mkdir -p build-asan && cd build-asan cmake .. -DCMAKE_BUILD_TYPE=Debug \ -DCMAKE_CXX_FLAGS="-fsanitize=address,leak -fno-omit-frame-pointer -g -O1" \ -DCMAKE_EXE_LINKER_FLAGS="-fsanitize=address,leak" cmake --build . -- -j4 # Executar com ASAN ASAN_OPTIONS=detect_leaks=1:halt_on_error=0:log_path=asan.log \ LSAN_OPTIONS=verbosity=1:log_threads=1 \ ./tfs

⚙️ Opção 2: CMakeLists.txt com opção de build

# Adicione no CMakeLists.txt do TFS option(ENABLE_ASAN "Enable AddressSanitizer" OFF) if(ENABLE_ASAN) if(CMAKE_CXX_COMPILER_ID MATCHES "GNU|Clang") target_compile_options(${PROJECT_NAME} PRIVATE -fsanitize=address,leak -fno-omit-frame-pointer -g -O1 ) target_link_options(${PROJECT_NAME} PRIVATE -fsanitize=address,leak ) endif() endif()
VariávelValorDescrição
ASAN_OPTIONSdetect_leaks=1Ativa LSAN integrado ao ASAN
ASAN_OPTIONShalt_on_error=0Continua após erro (coleta mais dados)

🔄 Workflow Completo — Do Zero ao Fix

1. Compile com símbolos de debug

cmake -DCMAKE_BUILD_TYPE=Debug -DCMAKE_CXX_FLAGS="-g -O0 -fno-omit-frame-pointer"

2. Execute com Valgrind ou ASAN

ASAN para iterações rápidas. Valgrind para análise profunda de leak persistente.

3. Reproduza o leak

Login/logout, spawns de monsters, compras no NPC, scripts Lua. Depois shutdown.

4. Analise o log

Foque em definitely lost. Identifique qual .cpp está na call stack.

5. Corrija o código

Use std::unique_ptr, std::shared_ptr, ou adicione o delete faltante.

6. Valide a correção

Recompile e execute novamente. O leak não deve mais aparecer no relatório.

🚫 Suppressions — Ignorar Falsos Positivos

Crie um arquivo tfs.supp para suprimir leaks de bibliotecas externas:

# Lua state global (still reachable, normal) { lua_state_global Memcheck:Leak match-leak-kinds: reachable ... fun:lua_newstate } # OpenSSL one-time init { openssl_init Memcheck:Leak match-leak-kinds: reachable,possible ... fun:CRYPTO_malloc fun:OPENSSL_init_crypto } # Boost.Asio thread pool { boost_asio_thread Memcheck:Leak match-leak-kinds: reachable ... obj:*libboost_thread* } # MySQL connector init { mysql_connector_init Memcheck:Leak match-leak-kinds: reachable,possible ... fun:mysql_server_init }
💡
Gerando automaticamente

valgrind --gen-suppressions=all --leak-check=full ./tfs 2> raw.supp — edite para manter apenas falsos positivos.

🩹 Correções Comuns no TFS

❌ Raw pointer sem delete → ✅ Smart pointer

// ANTES (leak) Monster* m = new Monster(mtype); creatures[m->getID()] = m; // se removeCreature() não chamar delete → leak // DEPOIS (correto) auto m = std::make_unique<Monster>(mtype); creatures[m->getID()] = std::move(m); // destruído automaticamente ao remover do map

❌ Exception antes do delete → ✅ RAII

// ANTES (leak em exceção) Item* item = new Item(itemId); processItem(item); // pode lançar exceção! delete item; // nunca executado se throw // DEPOIS (exception-safe) auto item = std::make_unique<Item>(itemId); processItem(item.get()); // se throw, unique_ptr destrói
✅
Regra de Ouro

Em C++ moderno: nunca use new raw sem smart pointer. Use std::make_unique para ownership exclusivo e std::make_shared para compartilhado.
